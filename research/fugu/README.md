# Fugu / TRINITY — an independent reproduction

A small, honest validation of **Sakana AI's Fugu** "learned model orchestration"
claim — the idea that a tiny model can route a query across a pool of frontier
LLMs and beat any single one of them. Reference paper: **TRINITY: An Evolved LLM
Coordinator** (arXiv:2512.04695). Reference open-source reimplementation:
[trotsky1997/openfugu](https://github.com/trotsky1997/openfugu) (Apache-2.0).

Everything here runs on **CPU with only numpy** in a few seconds. It is rendered
as a report on the homepage at **`/fugu`**.

> Not affiliated with Sakana AI. This is a personal reproduction by 大雷.

## What Fugu/TRINITY actually is

- A **0.6B** Qwen3 backbone produces a hidden state for the query.
- A **bias-free linear head** (the full evolved vector is `19456` numbers =
  `9×1024` SVF singular-value offsets + `10×1024` head rows) scores **7 worker
  slots** and **3 roles** (Worker / Thinker / Verifier).
- The top-scoring worker answers; **no worker weights are touched**.
- The router is trained **gradient-free** with **sep-CMA-ES** over that vector —
  not backprop. It is claimed to converge in only a few generations and still
  beat the best single model.

## What this harness checks

### Part B — mechanism (default, fully self-contained)
Reproduces the central claim on a synthetic but non-trivial pool: 7 specialist
workers (each strong on 2 of 5 task categories, weak elsewhere, so no single one
dominates). A linear router picks a worker by argmax over the query's hidden
vector; we train it with a **from-scratch sep-CMA-ES** to maximise held-out
reward, then compare to baselines.

Measured (seed 0; 8-seed summary in `results.json`):

| Strategy | Mean reward | vs best single |
|---|---|---|
| Random routing | 0.434 | −11% |
| Best single worker | 0.489 | — |
| **Learned router (sep-CMA-ES)** | **0.874** | **+79%** |
| Oracle (per-query best) | 0.874 | +79% |

Across **8 seeds**: **+79.0%** mean lift over the best single worker
(range +55.8% … +92.2%), **99.9%** of the oracle, converging in **~2
generations** — corroborating the paper's "few generations" and "beats the best
single model" claims in a controlled setting.

### Part A — control flow (opt-in)
Drives the **real `Coordinator`** loop from openfugu's `mini.py` with a scripted
router + mock worker pool (no weights, no API keys), asserting **6/6** documented
`step_trinity` behaviours: ACCEPT termination, cold-verifier re-routing, the raw
`verifier_no_response` path, the Thinker `<suggested_role>` override, and
solver-only `<reference_thought_N>` observation growth.

## Run it

```bash
pip install numpy
python research/fugu/validate.py            # Part B (self-contained)

# opt-in Part A — you fetch openfugu/mini.py yourself, then point at it:
python research/fugu/validate.py --mini /path/to/openfugu/mini.py
```

Part A is **opt-in by design**: this repo never downloads or executes
third-party code on your behalf. You supply the path to a local `mini.py`.

## Honest scope

**Reproduced:** the orchestration control flow (against real code) and the
"ES learns a router that beats the best single model" mechanism, including
few-generation convergence.

**Not done here:** running the real Qwen3-0.6B + `model_iter_60.npy` checkpoint
(needs GPU + the released weights), SWE-Bench / TerminalBench numbers (need a
live frontier-model pool / API keys), and the 7B Conductor / Fugu-Ultra DAG
variant. The synthetic pool isolates the routing mechanism from real-model noise;
it shows the mechanism is sound, not the production "+107%" figure. No API keys,
weights, or private data are used or committed.
